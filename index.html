0.0 — The problem

Your team's context lives in nobody's head.

Every week your team makes product decisions: "we defer dark mode to v3", "we chose Paddle over Stripe", "no session tokens, use JWT rotation". These decisions live in Slack threads, design docs, and people's heads.

Three months later, a contributor opens a PR that adds dark mode. The maintainer has to: remember the decision, find the rationale, write a polite rejection, and hope the same thing doesn't happen next week.

Existing OSS bots don't help here. Greptile reviews code. Dependabot bumps versions. Nobody reads your context.

repo-gardener reads your context tree — the knowledge base where those decisions live — and comments on PRs and issues that touch them, with specific tree node references.

How it works

Five steps. One comment per PR.

STEP 0

Load config

Read .claude/gardener-config.yaml for target repo and tree URL.

STEP 1

Scan

Fetch open PRs and issues via gh api. No clone.

STEP 2

Check state

Skip items already reviewed at the same commit. Honor @gardener pause/ignore.

STEP 3

Pull tree

Shallow clone the context tree from tree_repo.

STEP 4

Classify & comment

Map the diff to tree nodes. Pick a verdict. Edit existing comment in place.

Comment-only. Never pushes code. Never opens PRs. Never modifies branches. Idempotent across runs via commit-pinned state markers.

Five verdict types

Machine-readable. Human-scannable.

Every comment starts with a structured prefix: gardener:verdict: TYPE · severity: level. CI scripts can grep it. Maintainers see the verdict at a glance.

✅ ALIGNED
Fully consistent with tree decisions. Silent unless severity is high.
🆕 NEW_TERRITORY
Tree has no guidance. Safe but worth documenting.
❓ NEEDS_REVIEW
Partial overlap with tree; human judgment required.
⚠️ CONFLICT
Directly contradicts a tree decision. Recommend reconsideration.
❔ INSUFFICIENT_CONTEXT
Tree has partial info but not enough to decide.
Example output

A real gardener comment.

A PR adds a dark-mode toggle, but the team decided in 2024-01 to defer dark mode to v3. Gardener spots the conflict, cites the exact tree node, and recommends an action.

🌱 gardener:verdict: CONFLICT · severity: high · commit: abc1234

> [!WARNING]
> Context Review — checks product-context fit, not code
> correctness. Run Greptile/CodeRabbit for code review.

### Summary
This PR conflicts with the 2024-01 decision to defer dark mode to v3.

| Area | Item intent | Tree guidance | Fit |
|------|-------------|---------------|-----|
| Dark mode | Adds toggle | Deferred to v3 per design/no-dark-mode.md | ⚠️ Conflict |

### Tree nodes referenced
- design/no-dark-mode.md — team decision 2024-01
- roadmap/2024-q3.md — listed under "not in scope"

The first line is an HTML comment — invisible, machine-readable. The verdict line is visible at the top so maintainers see CONFLICT · high at a glance. The footer (omitted here) links back to repo-gardener and the tree repo so curious readers can learn what's reviewing them.

Two install modes

Review your repo. Or someone else's.

Gardener decouples "which repo to review" from "where to store the config". Same review logic, two install paths.

Mode 1

🔧 Maintainer mode

You maintain a repo. Install gardener in it. Commits land on your default branch. gardener:reviewed labels keep ALIGNED items silent.

target_repo: acme/api
tree_repo:   github.com/acme/context-tree
Mode 2

🌍 External reviewer mode

You want to review an OSS project you don't maintain. Store config in your own repo (typically your tree). Gardener never touches the target — only gh api.

target_repo: paperclipai/paperclip
tree_repo:   github.com/you/paperclip-tree
config_repo: you/paperclip-tree

No fork, no install PR on a stranger's repo, no trust negotiation. The maintainers get free context reviews from your GitHub identity.

vs other OSS bots

An empty lane.

Every other OSS bot competes on code-level review. None read context trees. Install gardener alongside Greptile or CodeRabbit — they don't overlap.

FAQ

Common questions

Does gardener push code or open PRs?
No. Comment-only. Always. Code-level fixes are someone else's job (Greptile, CodeRabbit, you).
What's a context tree?
A separate GitHub repo holding your project's design decisions, conventions, and constraints in markdown. First-Tree is the companion CLI for building one if you don't have one yet.
Will gardener spam my PRs?
No. Silent when ALIGNED + low severity (uses a label instead of a comment on your own repos). Re-reviews edit the existing comment in place — never stack. And maintainers can @gardener pause any item.
Does it conflict with Greptile / CodeRabbit?
No. Different lane. Greptile reviews code correctness; gardener reviews context fit. Install both. Each comment has a structured gardener:verdict: prefix so you can filter or grep.
Can I use it on a repo I don't maintain?
Yes. External reviewer mode (Mode 2 above). Store config in your own repo, target the upstream OSS project. Gardener uses your GitHub identity to comment.
How is state tracked?
HTML comment markers inside gardener's own comments (commit-pinned), plus a gardener:reviewed label for silent-aligned items, plus an eyes reaction as a soft lock. No separate database.
Where does it run?
Two ways: /loop 10m on your local machine via Claude Code, or /schedule as a cloud agent that runs every hour even when your laptop is closed.
Is it free?
Apache 2.0. The runtime cost is your Claude API tokens (the agent uses them to read tree nodes and write reviews).
